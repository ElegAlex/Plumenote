## 1. Filtrage Initial (Kill List)

### Contraintes bloquantes appliquées (P0) :

- **C1** : Self-hosted obligatoire — aucune donnée ne sort de l'infrastructure locale
- **C2** : Déployable en Docker, ressources raisonnables (cible ≤4 Go RAM, ≤3 containers)
- **C3** : Accès depuis poste utilisateur lambda sans compte admin
- **C4** : Licence compatible secteur public français (SILL / logiciel libre au sens DINUM)

### Éliminées — Non-viables comme solution principale :

- ❌ **#0 — Statu Quo** : 90-125h/jour perdues, 23 000-32 000h/an évaporées (P2.3). Inacceptable.
- ❌ **#3 — La Page d'Or** : Ne résout ni recherche, ni fraîcheur, ni import. Palliatif, pas une solution.
- ❌ **#6 — Outline Express** : Licence BSL 1.1 ≠ logiciel libre (C4). Non SILL. LDAP absent → Keycloak → 5-6 containers (C2). Guest/export PDF payants.
- ❌ **#7 — Docmost Deploy** : LDAP, import Word, API = Enterprise payant. Community = email/mot de passe uniquement → inutilisable en environnement AD. Pré-v1.0. Non SILL.
- ❌ **#9 — Wiki.js 3.0** : Projet en fin de vie. V3 en alpha depuis octobre 2022, aucune date de bêta. Communauté abandonnée.
- ❌ **#10 — PlumeNote MVP Minimaliste** : Renonce définitivement au graphe, au CRDT et aux supertags. Scope validé nulle part — le sponsor a validé le séquençage en 3 jalons (P2.1), pas la suppression. Différenciation vs BookStack insuffisante pour justifier un from scratch.
- ❌ **#14 — PlumeNote Vision (modèle équipe)** : Scope correct, modèle irréaliste (3-4 devs, 150-250k€/an). Aucun staffing disponible (P0). Le #23 porte le même scope avec le modèle réel.
- ❌ **#13 — Cerveau DSI (RAG)** : Couche d'augmentation, pas un point d'entrée documentaire unique. Combinable, pas autonome.

### Éliminées — Compléments, pas des solutions autonomes :

- ❌ **#4 — Concierge Humain** : Pas une solution principale. **Retenu comme accompagnement obligatoire.**
- ❌ **#2, #5 — Loupe Magique, Wrapper Chrome** : Ne résolvent que la recherche. Briques de transition.
- ❌ **#11 — BookStack++ (Frankenstein)** : Fork PHP/Laravel pour remplacer MySQL FT par Meilisearch = effort disproportionné, dette technique.
- ❌ **#12 — Hub Fédéré** : Pas d'unification réelle, pas de cycle de vie.
- ❌ **#15 à #22 — Famille IA & Analogies** : Philosophies de design ou modules combinables, pas de point d'entrée unique.

## 2. Matrice de Décision Pondérée (Top 4 Survivants)

### Justification des poids

- **Impact P2.3 (40%)** : Poids dominant, relevé de 35% à 40%. Le problème coûte 90-125h/jour. La raison d'être du projet est de résoudre les 6 items IN SCOPE. Si une solution ne résout pas le problème, rien d'autre ne compte.
- **Facilité d'usage (25%)** : P1.4 Insight #6 — "L'UX n'est pas une feature, c'est le business model." Adoption = 60% MAU à M+6. Inchangé.
- **Coût / Budget (15%)** : Budget = 200€/mois Claude Max + temps d'Alexandre. Pas de prestataire. Relevé de 10% car le coût d'opportunité du temps d'Alexandre est réel.
- **Simplicité Technique (10%)** : Abaissé de 15% à 10%. L'Infra a conditionné son adhésion à la facilité de maintenance, mais P0 dit "pas de deadline → qualité-first". La simplicité est un critère de confort, pas de survie.
- **Vitesse / Time-to-Value (10%)** : P0 dit "pas de deadline imposée". BookStack en alpha (pas en prod) donne de la marge (P2.1). Inchangé.

### Justification des notes

**BookStack Full Send (#1)** :

- _Impact 5/10_ : Recherche partielle (MySQL FT, pas d'indexation PJ, stemming FR limité). Accès universel ✅ (rôle Public). Contribution ✅ (meilleur WYSIWYG du lot). Fraîcheur ❌ (aucun signal, issue #5448 ouverte depuis des années, Dan Brown ne l'implémentera pas). Import Word ❌ (hack mammoth.js rudimentaire). Recherche PJ ❌. Graphe/supertags ❌. Couvre ~60% du P2.3 — les 40% manquants sont les plus critiques et n'ont aucune chance d'être comblés.
- _UX 8/10 (démontrée)_ : Unanimement le plus intuitif pour non-tech. Draw.io intégré. -2 : pas de blocs, pas de Cmd+K, pas de keyboard-first, architecture 4 niveaux rigide.
- _Coût 10/10_ : 0€ + ~5€/mois hébergement.
- _Simplicité 9/10_ : 2 containers, ~1-2 Go RAM, MIT, LDAP natif, SILL. -1 : LDAP nécessite 2-4h de débogage.
- _Vitesse 9/10_ : Déjà en alpha chez Lilian. Extension DSI en 2-3 semaines.

**Fork Augmenté Docmost (#8)** :

- _Impact 6/10_ : Base Docmost couvre recherche PG FTS, accès web, contribution TipTap. Le fork ajoute le graphe. Mais : LDAP à réimplémenter from scratch (code EE propriétaire, non copiable). Import Word idem. Fraîcheur non couverte.
- _UX 7/10 (base démontrée, fork incertain)_ : TipTap + Excalidraw/Mermaid. UX Notion-like. -1 vs BookStack car la base est démontrée mais le fork introduit de l'incertitude (l'UX peut diverger de l'upstream).
- _Coût 7/10_ : 0€ licence mais ~3 mois de temps d'Alexandre.
- _Simplicité 4/10_ : Fork AGPL + suivi upstream + LDAP/import/recherche PJ à réimplémenter. Stack NestJS complexe.
- _Vitesse 5/10_ : ~3-4 mois avant production.

**PlumeNote Vision Séquencée (#23)** :

- _Impact 9/10_ : Seule option couvrant les 6 IN SCOPE + vision long terme. J1 : Meilisearch <3s + accès universel Go + signaux de fraîcheur natifs (LE différenciateur absent du marché — P3.2) + import Pandoc + TipTap <5 min. J2 : graphe AGE + backlinks (Opportunité #4, P1.2 "très élevée" — personne en self-hosted). J3 : supertags + objets typés (avance concurrentielle unique). -1 : conditionnel à la livraison (spike P2.2 non exécuté).
- _UX 5/10 (projetée, non démontrée)_ : L'ambition est la plus élevée (Linear-inspired, Cmd+K, <100ms, blocs, keyboard-first). Le sponsor connaît Obsidian/Linear/Tana et construit lui-même. Mais zéro utilisateur n'a vu le produit. L'étude METR (2025) montre -19% de productivité réelle pour le code IA-assisté malgré +20% de perception — l'UX finale sera probablement en dessous de la vision. Un 5 reflète honnêtement : potentiel élevé (8-9), probabilité de livraison au niveau visé (6-7), réalité actuelle (0).
- _Coût 8/10_ : 200€/mois Claude Max + temps d'Alexandre. Pas de recrutement. Viable tant que la vélocité tient.
- _Simplicité 3/10_ : 5 technos au J1 (React+TipTap+Go+PostgreSQL+Meilisearch), AGE au J2, supertags au J3. Bus factor = 1. P2.2 (H8) identifie l'assemblage comme risque majeur. Veracode 2025 (45% de vulnérabilités dans le code IA-assisté) impose un audit sécurité avant mise en prod → complexité additionnelle. Abaissé de 4 à 3 par rapport aux versions précédentes.
- _Vitesse 3/10_ : 8-10 semaines pour le J1 SI le spike passe. ~6-9 mois pour la vision complète. L'étude METR (-19% productivité réelle) suggère que les estimations vibe coding sont optimistes. Abaissé de 4 à 3.

**XWiki Deploy (P3.2)** :

- _Impact 7/10_ : Seul outil cochant presque toutes les cases fonctionnelles : import Office natif ✅, Solr indexant les PJ ✅, LDAP mature ✅, accès guest ✅, permissions granulaires ✅. -3 : fraîcheur pas dans le cœur gratuit (extension Pro ou XClass), UX datée freine la contribution, pas de graphe ni supertags.
- _UX 4/10 (démontrée, datée)_ : Skin Flamingo 2010, CKEditor. Cristal non prêt (deep research : "le but n'est PAS que les utilisateurs utilisent Cristal pour travailler"). P1.4 Insight #6 : "les techniciens jugent en 5 secondes."
- _Coût 7/10_ : LGPL gratuit. VM 8-16 Go = 30-50€/mois. Support Pro ~1 500€/an optionnel.
- _Simplicité 3/10_ : 8 Go RAM minimum → viole C2. Monolithe Java/Tomcat+Solr. OutOfMemoryError documentés. Configuration initiale : plusieurs jours.
- _Vitesse 7/10_ : 2-3 semaines de déploiement + configuration. Documentation abondante.

## 3. Analyse DVF des Finalistes

### Ce que dit la matrice — et ce qu'elle ne dit pas

La matrice produit **BookStack (#1) comme option au score le plus élevé (7.30)**. C'est un résultat mathématiquement correct : BookStack domine sur 4 critères sur 5 (UX, Coût, Simplicité, Vitesse). PlumeNote #23 ne le bat que sur un seul critère — l'Impact — mais c'est le critère le plus lourd (40%).

La matrice évalue des **options en isolation**. Elle ne compare pas des **stratégies** (combinaisons d'options déployées dans le temps). Or la question du dossier n'est pas "quel outil déployer maintenant" — c'est "quelle trajectoire maximise la valeur sur 9 mois dans un contexte où aucune solution du marché ne couvre le besoin complet (P3.2)".

La réponse est une stratégie, pas une option : **BUILD #23 comme cible + BUY #1 comme pont.**

### 🥇 Solution Cible : PlumeNote Vision Séquencée (#23) — Score Impact : 9/10 | Score global : 6.65

- **Pourquoi lui comme cible ?** : Le P3.2 conclut que le marché ne répond pas au besoin. BookStack couvre ~60%. XWiki couvre ~80% mais tue l'adoption. Aucun outil gratuit n'offre les signaux de fraîcheur (le "White Space" de P3.2). Aucun outil self-hosted multi-utilisateurs n'a de graphe d'exploration (Opportunité #4, P1.2). Le #23 est la seule option qui couvre les 6 IN SCOPE ET porte les 3 jalons de la vision P2.1. Le sponsor l'a validé, dispose du modèle de réalisation, et a commencé le développement. L'approche est encadrée par les gates P2.2.
    
- **Le trade-off** : Alignement maximal avec le problème, contrôle total sur la roadmap, signaux de fraîcheur immédiat (J1), graphe structural (J2), supertags unique (J3) — en échange d'un risque d'exécution élevé (bus factor = 1, code IA-assisté à auditer, 6-9 mois de développement) et d'une UX qui reste une promesse tant que le produit n'est pas entre les mains de Didier.
    
- **Risque principal — bus factor = 1** : P2.2 qualifie ce risque de "mortel". Alexandre est simultanément sponsor, développeur, chef de projet et décideur. Toute indisponibilité arrête le projet. Mitigation : si jalon 1 > 10 semaines → évaluation du fork Docmost (#8). Documentation de l'architecture au fil de l'eau.
    
- **Risque secondaire — qualité du code IA-assisté** : Veracode 2025 rapporte 45% de vulnérabilités dans le code IA-assisté. Pour un déploiement secteur public (RGPD, potentiellement RGS), un audit de sécurité dédié est non négociable avant la mise en production.
    
- **Question ouverte — les jalons 2 et 3 sont-ils réalistes pour une personne seule ?** Le jalon 1 (éditeur + recherche + permissions + import) est du CRUD augmenté — le terrain le plus favorable au vibe coding. Le jalon 2 (Apache AGE, requêtes Cypher, visualisation force-graph) et le jalon 3 (supertags, modélisation d'objets typés, vues dynamiques) sont des problèmes de modélisation et d'intégration d'un autre calibre. Le P2.2 (H8) le dit : "L'IA assiste mais ne remplace pas la compréhension architecturale." Si le jalon 1 réussit mais que les jalons 2-3 s'enlisent, le produit livré ressemblera au #10 (MVP Minimaliste) — celui qu'on a éliminé pour différenciation insuffisante. Ce risque doit être surveillé jalon par jalon. Un PlumeNote bloqué au jalon 1 n'est pas la vision du sponsor — c'est un BookStack custom sans communauté.
    

### 🥈 Pont Immédiat + Plan B : BookStack Full Send (#1) — Score global : 7.30

- **Pourquoi lui en pont ?** : PlumeNote #23 n'existera pas avant 8-10 semaines minimum. BookStack est déjà en alpha chez Lilian, déployable en 2-3 semaines, 0€, 2 containers, LDAP natif, SILL. Il résout ~60% du problème immédiatement et sert trois objectifs : (1) délivrer de la valeur pendant le spike, (2) valider l'appétit d'adoption, (3) constituer le patrimoine de contenu migrable vers PlumeNote via API.
    
- **Pourquoi aussi plan B ?** : Si le spike échoue ou si Alexandre est indisponible, BookStack est déjà en production. Il devient la solution permanente, enrichie par des scripts (Pandoc pour l'import, tags manuels pour la fraîcheur).
    
- **Ses limites comme solution permanente** : Fraîcheur ❌, import Word ❌, recherche PJ ❌, architecture rigide, bus factor = 1 (Dan Brown), aucune perspective de graphe ni supertags. Rester sur BookStack, c'est accepter de ne résoudre que 60% du problème — et de renoncer à la vision.

## 4. Recommandation BUILD vs BUY vs ADAPT

- [x] **BUILD** (PlumeNote Vision Séquencée #23 — solution cible)
- [x] **BUY transitoire** (BookStack #1 — pont immédiat + plan B)

### Stratégie : deux voies parallèles (Combo "Quick Win + Vision" de P3.1)

**Voie 1 — BookStack NOW (S+1 à S+3)** : Déployer BookStack à toute la DSI. Importer le patrimoine via Pandoc+API. Nommer un référent documentaire (#4). Tags manuels pour la fraîcheur (#19). Objectif : valeur immédiate + base de contenu migrable.

**Voie 2 — PlumeNote BUILD (S+1 → M+9, en parallèle)** :

|Semaine|Action|Gate (critère de passage)|
|---|---|---|
|S+1-2|Spike technique : TipTap + Meilisearch + Go + import Pandoc|Prototype fonctionnel qui affiche un résultat de recherche en <3s|
|S+2|Démo interne aux 4 contributeurs-clés|≥3 sur 4 déclarent vouloir l'utiliser|
|S+1-4|Concierge MVP contenu : 20 fiches de procédures critiques|≥15 fiches rédigées|
|**S+5**|**Gate Review**|**3/3 tests passés → 🟢 GO BUILD FRANC**|
|S+5-10|Jalon 1 : éditeur + recherche + permissions + fraîcheur + import|Produit déployable, ≥5 utilisateurs actifs|
|M+3|Bascule : migration contenu BookStack → PlumeNote via API|BookStack décommissionné|
|M+3-5|Jalon 2 : graphe AGE + backlinks + force-graph|Cartographie navigable, ≥1 vue croisée fonctionnelle|
|M+5-9|Jalon 3 : supertags + objets typés + vues dynamiques|Modélisation serveurs↔applis↔habilitations opérationnelle|

### Plan d'escalade (critères explicites)

|Si...|Alors...|Qui décide ?|
|---|---|---|
|Spike échoue (S+2) : prototype non fonctionnel|Abandon du from scratch. Évaluation du Fork Docmost (#8) pendant 2 semaines. BookStack reste en prod.|Alexandre (sponsor)|
|Jalon 1 > 10 semaines|Évaluation du fork Docmost (#8). Si fork non convaincant en 2 semaines → BookStack accepté comme solution permanente.|Alexandre + Mathieu Messe|
|Jalon 2 échoue (AGE trop complexe solo)|PlumeNote reste au jalon 1 (= fondamentaux sans graphe). Le graphe est réévalué quand/si un renfort dev se matérialise.|Alexandre|
|Alexandre indisponible > 2 semaines|Gel du développement. BookStack continue. Reprise sur base de la documentation d'architecture.|Mathieu Messe (backup)|
