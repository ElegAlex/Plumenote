## 1. Top Solutions du Marché (Analyse critique)

### 🏆 Solution A : BookStack

- **Type** : Open Source (MIT) — Développeur solo (Dan Brown, UK)
- **Promesse** : _"Documentation simple et gratuite"_ — wiki WYSIWYG intuitif pour non-techniques
- **Réalité (Ce que ça fait vraiment)** : Un wiki structuré en 4 niveaux rigides (Étagères → Livres → Chapitres → Pages) avec un éditeur WYSIWYG (TinyMCE → migration Lexical en cours) qui est objectivement le plus agréable du lot pour des contributeurs non-tech. Draw.io natif intégré. Recherche full-text MySQL/MariaDB basique. API REST complète. Déjà en alpha dans la DSI CPAM (champion interne : Lilian).
- **Pricing Estimé** : Gratuit (MIT, 100% FOSS). Coût réel = hébergement (~5€/mois VPS) + temps humain de customisation.
- **🟢 Le Bon** :
    - Meilleur rapport simplicité/efficacité du marché : 2 containers Docker, 1 Go RAM, installation en 15 minutes
    - UX contributeur unanimement saluée — le seuil de contribution est le plus bas du comparatif
    - API REST complète et bien documentée — tout est scriptable
    - Licence MIT = liberté totale de fork, modification, redistribution
    - Accès guest natif et gratuit (rôle Public configurable)
    - LDAP natif (avec frictions, cf. ci-dessous)
    - Draw.io intégré = cartographies techniques sans outil tiers
- **🔴 Le Mauvais** :
    - **Aucun import Word/PDF natif** — un hack communautaire mammoth.js existe mais la conversion est décrite comme "relatively simplistic, you'll likely lose layout or formatting" avec des images base64 qui "cause editor performance to be particularly slow". Pour migrer des centaines de procédures Word, il faut développer un pipeline custom via API.
    - **Aucun signal de fraîcheur du contenu** — pas de date de révision, pas de rappel, pas d'expiration, pas de workflow d'approbation. Issue GitHub #5448 ouverte depuis des années, sans réponse du mainteneur. Bloquant pour la conformité documentaire CPAM.
    - **Bus factor = 1** — Dan Brown, seul développeur, vit de £45 500/an de sponsors. Il a confié sur Noted.lol : _"The sheer weight of requests within the GitHub issues list has put me close to giving up on the project a few times for my own sanity."_ La licence MIT permet un fork, mais le savoir institutionnel repose sur une personne.
    - **Recherche MySQL limitée en français** — pas de stemming FR, problèmes d'accents documentés, mot minimum 3 caractères, pas d'indexation des pièces jointes PDF.
    - **Architecture 4 niveaux rigide** — Étagères → Livres → Chapitres → Pages, pas extensible. Si la structure documentaire ne colle pas à ce modèle, on force.
    - **LDAP : 2-4h de débogage** à l'installation (piège du double `$$` dans docker-compose, erreurs silencieuses avec LDAPS).
    - **Permissions sans tableau de bord** — un utilisateur GitHub (#1736) : _"Currently one has to check the rights/permissions book by book, chapter by chapter, page by page… This is a security nightmare!"_
- **💬 Verbatim Utilisateur (Synthétisé)** : _"Great search, WYSIWYG OR Markdown supported, great integration into Diagrams.net. Thank you, thank you, thank you."_ (HN) — mais aussi : _"Importing Word makes the difference in whether or not BookStack is the right tool. Manual copy & paste is not a practical workaround."_ (GitHub #4258)

### 🏆 Solution B : XWiki

- **Type** : Open Source (LGPL 2.1) — Entreprise française (XWiki SAS, Paris, ~74 employés, 20 ans)
- **Promesse** : _"Le wiki enterprise open source"_ — alternative à Confluence pour le secteur public
- **Réalité (Ce que ça fait vraiment)** : Le seul candidat qui répond nativement à TOUTES les exigences techniques critiques du cahier des charges CPAM : import Office natif (LibreOffice intégré), recherche Solr indexant les pièces jointes (PDF, Word), LDAP/AD mature avec extension dédiée, permissions par page/espace/sous-wiki, API REST + 900 extensions, collaboration temps réel (CKEditor), workflow d'approbation possible via XClasses/extensions Pro. En contrepartie, c'est un monolithe Java lourd (8 Go RAM minimum) avec une UX datée.
- **Pricing Estimé** : Gratuit (Community LGPL). XWiki Pro (support + extensions premium) : à partir de ~1 500 €/an pour 25 users. Coût réel = VM 8-16 Go RAM (~30-50€/mois) + temps de configuration conséquent.
- **🟢 Le Bon** :
    - **Entreprise française pérenne** — XWiki SAS, ~74 employés, autofinancée, 20 ans d'existence, pas de dépendance VC. Références secteur public : CNFPT (WikiTerritorial), INSEE, SNCF, Naval Group, EDF. Fait partie d'openDesk (suite souveraine du gouvernement allemand).
    - **Import Office natif** — Word → pages wiki éditables (texte, images, tableaux), Excel → tableaux, PowerPoint → galerie images. Taux de succès >99% sur migrations massives. C'est LE différenciateur pour migrer un patrimoine Word existant.
    - **Recherche Solr intégrée** — seul outil du comparatif à indexer le contenu des pièces jointes (PDF, Word). Amazon utilise XWiki avec "millions de pages".
    - **LDAP/AD mature** — extension Active Directory dédiée, mapping de groupes, synchronisation automatique, documentation extensive.
    - **Permissions très granulaires** — page, espace, sous-wiki. Le système le plus fin du comparatif.
    - **XClasses** — mécanisme unique permettant d'ajouter des champs structurés (date de révision, propriétaire, statut) à n'importe quelle page sans développement Java. C'est le chemin vers les signaux de fraîcheur.
    - **900+ extensions** — écosystème riche, marketplace active
- **🔴 Le Mauvais** :
    - **UX objectivement datée** — skin Flamingo (Bootstrap) des années 2010, CKEditor correct mais pas au niveau de BookStack/Outline. Un projet "Cristal" de refonte frontend est en cours mais pas encore livré. Pour l'adoption par des agents non-DSI, c'est le risque principal.
    - **Lourd en ressources** — 8 Go RAM minimum (JVM 4 Go + Solr + PostgreSQL + OS). Rapports d'OutOfMemoryError récurrents en Docker. Nécessite un serveur/VM dédié conséquent.
    - **Permissions puissantes mais complexes** — la règle de denial implicite est source de confusion. Un utilisateur confesse : _"Access rights driving me mad… I just cannot find my way to the access rights system."_
    - **Courbe d'apprentissage admin** — la configuration initiale (droits, skin, LibreOffice, Solr) demande un investissement de plusieurs jours.
    - **Signaux de fraîcheur = payant ou custom** — pas dans le cœur gratuit. L'extension Pro "Procedures" (payante) ou un développement XClass custom sont nécessaires.
- **💬 Verbatim Utilisateur (Synthétisé)** : _"XWiki (avec Keycloak) sont de loin l'équipe la plus sérieuse et efficace parmi les projets open source que nous intégrons."_ (Jérôme Courat) — mais aussi : _"The user interface, while functional, may not be as modern or intuitive as some newer wiki platforms."_ (comparatifs multiples)

### 🏆 Solution C : Outline

- **Type** : Source disponible (BSL 1.1 — **pas open source au sens OSI**) — Petite entreprise (General Outline Inc.)
- **Promesse** : _"La base de connaissance la plus rapide pour les équipes"_ — UX Notion-like
- **Réalité (Ce que ça fait vraiment)** : Un wiki bloc-editor avec l'UX la plus moderne du marché (slash commands, collab temps réel, mode sombre, backlinks). Mais un cauchemar à self-hoster (6+ containers : app + PostgreSQL + Redis + S3 + OAuth provider obligatoire) et un modèle commercial qui verrouille des fonctions basiques (guest, export PDF) derrière l'édition payante.
- **Pricing Estimé** : SaaS 10$/user/mois. Self-hosted : "gratuit" mais nécessite un provider OAuth externe obligatoire (pas de login email/mot de passe natif). Business/Enterprise : pricing non public.
- **🟢 Le Bon** :
    - UX la plus séduisante du lot — éditeur Notion-like, collaboration temps réel native (ProseMirror + Yjs)
    - API RPC complète avec webhooks
    - Import Word (.docx) fonctionnel par drag-and-drop
    - 37 400+ ⭐ GitHub — communauté active
- **🔴 Le Mauvais** :
    - **Licence BSL 1.1 — PAS open source**. Compatibilité incertaine avec les exigences DINUM/SILL du secteur public français. Un utilisateur HN résume : _"What an ambiguous mess."_
    - **LDAP : NON SUPPORTÉ**. Nécessite un bridge OIDC (Keycloak/Authentik) avec sa propre base PostgreSQL. Résultat : 6+ containers en production. _"Outline is probably the most complex docker container you will deploy."_
    - **Guest et export PDF = payants** — édition Business/Enterprise uniquement. Verrouillage de fonctions basiques.
    - **Pas de mapping groupes LDAP** — même avec un bridge (issue #8168 ouverte)
    - **Permissions limitées** — 3 rôles seulement (Viewer, Editor, Admin). Insuffisant pour 50 personnes avec cloisonnement par service.
    - **Import PDF impossible**
    - **Aucun signal de fraîcheur**
- **💬 Verbatim Utilisateur (Synthétisé)** : Un collègue utilisateur a demandé si c'était _"some weird version of Notion"_ — mais aussi : _"After trying and failing many times…"_ (auto-hébergement)

### 🏆 Solution D : Docmost

- **Type** : Open Source (AGPL-3.0) — Développeur solo (Philip Okonkwo, @Philipinho)
- **Promesse** : _"Wiki enterprise-ready open source — alternative à Notion/Confluence"_
- **Réalité (Ce que ça fait vraiment)** : Un wiki moderne (éditeur TipTap, collab temps réel, Draw.io/Excalidraw/Mermaid intégrés) mais encore en pré-v1.0 (v0.25.3 en mars 2026). Les fonctions enterprise critiques (LDAP, API, import) sont verrouillées dans l'édition payante.
- **Pricing Estimé** : Gratuit (Community AGPL). Enterprise : pricing non public (devis requis). Le free tier est un "jouet" — inutilisable en environnement AD.
- **🟢 Le Bon** :
    - Éditeur TipTap moderne avec collab temps réel
    - Draw.io + Excalidraw + Mermaid intégrés (bon pour la doc technique)
    - Déploiement Docker simple (3 containers)
    - 19 300+ ⭐ en <3 ans — momentum fort
    - Licence AGPL (vrai open source, contrairement à Outline)
- **🔴 Le Mauvais** :
    - **Pré-v1.0** — pas production-ready pour un usage critique. Le cabinet WZ-IT l'a écarté pour _"still younger project status"_.
    - **LDAP/SSO = Enterprise payant uniquement** — Community = email/mot de passe seulement. Inutilisable dans un environnement Active Directory.
    - **API REST = Enterprise payant**
    - **Import Word = Enterprise payant** — le Community n'importe rien.
    - **Permissions par espace seulement** — pas par page. Insuffisant.
    - **Aucun système de notifications** — pas d'alertes @mention, pas de suivi de page, pas de notification email (issue #1688).
    - **Aucun signal de fraîcheur**
    - **Bus factor ≈ 1** — pratiquement tout le développement repose sur @Philipinho. Les 47 contributeurs GitHub ne font que des corrections mineures.
    - **Documentation lacunaire** — un utilisateur : _"I ran into many issues, and had to rely on AI to correct your installation instructions."_
- **💬 Verbatim Utilisateur (Synthétisé)** : _"The biggest draw is the ability to self-host, which makes my data and knowledge mine."_ (XDA) — mais le cabinet WZ-IT a préféré BookStack pour la maturité.

## 3. Analyse des Patterns (UX & Tech)

### **Standard du marché (Must Have) — Ce que tout le monde fait :**

Toutes les solutions (sauf Wiki.js en déclin) proposent un **éditeur WYSIWYG riche** avec support Markdown, un **versionnement** de pages avec historique et rollback, une **recherche full-text** transversale, et un **déploiement Docker** documenté. **Si PlumeNote ne propose pas au minimum ces 4 piliers dès le jalon 1, il sera perçu comme inférieur à BookStack** — qui les coche déjà tous pour zéro euro. C'est la ligne de base non négociable.

Le marché converge aussi vers l'**éditeur bloc-based** (Outline, Docmost, AFFiNE) comme standard UX de la nouvelle génération. L'éditeur page-based classique (BookStack, XWiki) est perçu comme "fonctionnel mais daté". PlumeNote, en choisissant TipTap (bloc-based), est déjà aligné sur cette tendance.

### **Les "Anti-Patterns" (À éviter) :**

- **Le piège de la complexité de déploiement** : Outline est le cas d'école. Une UX parfaite ne sert à rien si l'admin sys met 3 jours à installer l'outil. 6+ containers + OAuth obligatoire = mort pour une DSI qui fait du run. PlumeNote doit rester sous les 3 containers Docker.
- **Le paywall sur les fonctions basiques** : Docmost et Outline verrouillent LDAP, API, guest, import derrière l'édition payante. Cela génère de la frustration et de la méfiance dans la communauté. PlumeNote (s'il reste internal) n'a pas ce problème, mais si jamais il est open-sourcé, toutes les fonctions CPAM-critiques doivent être dans le cœur gratuit.
- **Le bus factor = 1 sans plan de succession** : BookStack, Wiki.js, Docmost — trois projets populaires, trois projets à un commit d'un mainteneur de la disparition. C'est ironiquement le même risque que PlumeNote (P2.2, H5 — Alexandre développeur solo). Le pattern récurrent : tout va bien pendant les premières années, puis l'épuisement arrive (cf. BookStack : _"close to giving up"_).
- **L'absence de signaux de fraîcheur** : AUCUNE solution gratuite du marché ne propose de cycle de vie du contenu en standard. C'est un angle mort systémique du marché wiki open source — les outils gèrent la création et l'édition, jamais l'obsolescence. Pourtant, P1.4 identifie la fraîcheur perçue comme pain point #2 (confiance 90%).

## 4. Le "White Space" (Opportunité de Différenciation)

Le marché ne répond pas correctement au besoin **"recherche transversale + accès web universel + signaux de fraîcheur + coût de contribution minimal"** pour une DSI de taille moyenne (20-100 personnes) dans le secteur public français :

- **BookStack** coche simplicité et accès web, mais pas l'import du patrimoine ni la fraîcheur → **trop limité** sur les exigences documentaires.
- **XWiki** coche l'import, la recherche Solr, le LDAP et la pérennité, mais l'UX est datée et le déploiement lourd → **trop complexe** pour être adopté par des non-techniques sans accompagnement significatif.
- **Outline** a la meilleure UX mais n'est pas open source, pas LDAP, et paywalle des fonctions basiques → **incompatible secteur public**.
- **Docmost** est prometteur mais pré-v1.0 et paywalle le LDAP → **pas prêt**.
- **Wiki.js** est en fin de vie → **éliminé**.

👉 **Notre opportunité est ici :**

Un outil qui combine **l'UX contributeur de BookStack/Outline** (éditeur intuitif, déploiement ≤3 containers, <5 min pour rédiger une procédure) avec **la profondeur technique de XWiki** (recherche indexant les PJ, import du patrimoine Word/PDF, LDAP natif, permissions granulaires) et qui ajoute **ce qu'aucun outil du marché ne fait en standard gratuit** : les **signaux de fraîcheur du contenu** (date de révision, rappel automatique, badge d'obsolescence, workflow de validation). Le tout self-hosted, open source, en français, léger (<4 Go RAM).

C'est exactement le positionnement de PlumeNote décrit en P2.1 — mais **sans le graphe au jalon 1**. Le différenciateur immédiat n'est pas le graphe (besoin latent "Should Have" selon P1.3) mais la combinaison simplicité + fraîcheur + recherche + accès universel. Le graphe devient le "Wow" du jalon 2, une fois que les fondamentaux ont conquis l'adoption.

## 5. Recommandation BUILD vs BUY

### Verdict : aucune solution du marché ne couvre le besoin.

- [ ] **BUY** : Non.
- [x] **BUILD** : Oui — construire PlumeNote.

### Pourquoi pas BUY ?

**XWiki** est le seul outil techniquement complet (import Office, Solr, LDAP, permissions). Mais son UX datée (années 2010) tuera l'adoption. P1.4 (Insight #6) est sans appel : _"L'UX n'est pas une feature — c'est le business model. Les techniciens jugent en 5 secondes."_ Si Didier ouvre XWiki et que ça ressemble à 2010, il retourne demander au collègue d'à côté. La couverture fonctionnelle ne vaut rien sans adoption.

**BookStack** a la meilleure UX contributeur mais ne couvre que ~60% du besoin. Pas d'import Word, pas de fraîcheur, recherche limitée en français, bus factor = 1. Le transformer en outil complet coûterait autant de développement custom que de construire PlumeNote — sans le graphe ni l'architecture cible.

**Outline, Docmost, Wiki.js** sont éliminés : licence non libre (Outline), pré-v1.0 avec LDAP payant (Docmost), projet en fin de vie (Wiki.js).

### Ce que le benchmark valide pour PlumeNote

Le BUILD n'est pas un saut dans le vide. Le benchmark fournit un cahier des charges validé par le marché :

- **Éditeur TipTap** — standard de la nouvelle génération (Outline, Docmost, AFFiNE l'utilisent). Choix déjà acté, confirmé.
- **Recherche Meilisearch** — surpasse MySQL FT (BookStack) et évite la lourdeur de Solr (XWiki). Choix déjà acté, confirmé.
- **Import Word via Pandoc/mammoth.js** — comble le trou #1 de BookStack. Indispensable pour migrer le patrimoine existant.
- **LDAP natif** — copier le modèle XWiki (extension dédiée, mapping de groupes). Pas de bridge OIDC à la Outline.
- **Signaux de fraîcheur natifs** — c'est LE différenciateur. Aucun outil gratuit du marché ne le fait. Date de révision, rappel auto, badge d'obsolescence. C'est ce qui transforme PlumeNote d'un "wiki de plus" en un outil de gouvernance documentaire.
- **Accès guest par URL** — standard BookStack/XWiki. Condition P2.3 (accès depuis poste non-DSI).
- **≤3 containers Docker, <4 Go RAM** — ne pas reproduire le piège XWiki (8 Go) ni Outline (6+ containers).
- **Graphe = jalon 2, pas jalon 1** — confirmé par le benchmark : aucun concurrent self-hosted multi-user ne l'a. C'est un "Wow" futur, pas un prérequis.
