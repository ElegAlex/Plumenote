## 3. Les "Moonshots" & Approches Radicales

_(Innovation de rupture, IA, changement de paradigme)_

|#|Nom de l'option|Le concept fou|
|---|---|---|
|13|**"Le Cerveau DSI" (RAG interne)**|Ingérer toute la documentation DSI (Word, PDF, TXT, BookStack) dans un pipeline RAG (Retrieval-Augmented Generation) self-hosted. Le technicien pose sa question en langage naturel ("Comment configurer le VPN ProWeb ?") et le LLM local (Mistral/Llama via Ollama, déployé sur serveur Infra) répond avec la procédure exacte, sourcée, avec le lien vers le document d'origine et sa date de fraîcheur. Pas de navigation, pas de recherche, pas de wiki — juste une question → réponse. Interface : un chatbot web accessible depuis n'importe quel poste. L'IA ne remplace pas la documentation, elle la rend transparente.|
|14|**"PlumeNote Vision Complète (modèle équipe)"**|Le rêve du sponsor porté à son terme par une équipe dédiée : 3-4 développeurs (recrutement ou prestation) construisent PlumeNote tel qu'imaginé en P0 — graphe AGE + éditeur TipTap + CRDT Yjs + supertags + recherche Meilisearch + import multi-format + UX Linear-grade + analytics intégrées. Budget réel : 150-250k€/an pendant 18 mois. Résultat : un produit de niveau startup, potentiellement open-sourceable, qui dépasse tout ce qui existe en self-hosted.|
|15|**"L'Auto-Doc"**|Un agent IA qui écoute les résolutions d'incidents dans GLPI (tickets fermés avec résolution documentée) et génère automatiquement des brouillons de procédures. Le technicien qui ferme un ticket avec un commentaire technique de >5 lignes reçoit une notification : "J'ai transformé votre résolution en procédure — voulez-vous la publier ?" La contribution ne coûte plus rien — elle est un sous-produit du travail quotidien. Résout le pattern P1.4 : "le KM meurt par le coût de contribution".|
|16|**"Le Graphe Vivant" (Knowledge Graph automatisé)**|Sans outil de rédaction — un moteur d'extraction automatique (NLP/NER) qui parse toute la documentation existante et construit automatiquement un graphe de connaissances : entités (serveurs, applicatifs, procédures, personnes) et relations (hébergé_sur, documenté_par, maintenu_par). Stocké dans PostgreSQL+AGE. Visualisé dans un portail web force-graph. Le graphe d'Alexandre se construit tout seul à partir du patrimoine existant, sans migration manuelle.|

## 4. Les Solutions "Out-of-the-Box" (Analogies)

_(Inspiré d'autres industries)_

|#|Nom de l'option|Inspiré de|Concept transposé|
|---|---|---|---|
|17|**"Le Stack Overflow DSI"**|Stack Overflow / Teams|Plateforme Q&A interne : les techniciens posent des questions, les sachants répondent, les meilleures réponses sont votées et deviennent la documentation canonique. Les questions récurrentes (identifiées par fréquence) sont promues en procédures formelles. On utilise Stack Overflow for Teams (self-hosted) ou un équivalent open source (Answer, Discourse). La documentation émerge de l'usage au lieu d'être imposée top-down.|
|18|**"Le Netflix DSI"**|Netflix (recommandation personnalisée)|Un portail documentaire qui connaît ton rôle (SCI, Infra, Support) et te propose un "flux personnalisé" : les procédures les plus consultées par ton service cette semaine, les documents récemment mis à jour dans ton domaine, les nouvelles procédures ajoutées. Pas besoin de chercher — la doc pertinente vient à toi. Algorithme de recommandation basé sur le profil utilisateur + consultation des pairs du même service.|
|19|**"Le Tinder de la Fraîcheur"**|Tinder (swipe de validation)|Chaque semaine, le référent documentaire de chaque service reçoit 5 fiches à "swiper" : ✅ Toujours valide / ❌ Obsolète / 🔄 À mettre à jour. Le système gamifie la revue de fraîcheur : tableau de bord par service, score de "santé documentaire", mini-compétition entre services. 10 secondes par fiche. Le cycle de vie documentaire (pain point #2, confiance 90%) est résolu par un micro-effort distribué au lieu d'un macro-effort centralisé.|
|20|**"L'Uber de la Connaissance"**|Uber (mise en relation on-demand)|Quand le technicien ne trouve pas la réponse dans la documentation, au lieu d'interrompre un collègue au hasard, un système de routage intelligent l'oriente vers le sachant le plus pertinent ET le plus disponible (basé sur un graphe de compétences + calendrier). La conversation est enregistrée et transformée en fiche documentaire (cf. option #15). Le réseau informel est digitalisé, pas supprimé — et chaque interaction produit de la documentation.|
|21|**"Le GitHub de la Doc"**|GitHub (collaboration + pull requests)|La documentation est gérée comme du code : fichiers Markdown dans un repo Git, branches pour les modifications, "pull requests" pour les revues, CI/CD pour la publication automatique sur un site statique (MkDocs, Docusaurus, ou Hugo). Les techniciens qui maîtrisent Git (Infra, Dev) contribuent nativement. Les autres passent par une interface web simplifiée (éditeur Markdown en ligne type HackMD/CodiMD). Versionnement natif, historique complet, diff visuels.|
|22|**"La Radio DSI"**|Podcasts internes (Spotify for Work)|Les sachants enregistrent des "micro-capsules" audio/vidéo de 2-3 minutes sur leurs procédures les plus demandées (via un simple lien web + micro du PC). Les capsules sont transcrites automatiquement (Whisper) et deviennent des fiches documentaires indexées. Le coût de contribution passe de "rédiger un Word" (30-60 min) à "parler pendant 3 minutes" (<5 min tout compris). Adresse frontalement le pain point #8 (alimentation trop coûteuse).|

## 6. Synthèse : Les Familles de Solutions

_(Regroupement pour faciliter le choix en P3.2)_

- 🟢 **Famille A — "Process & Humain"** : Options #3, #4, #19 → _Peu de tech, beaucoup d'organisation._ Coût quasi nul, impact immédiat mais limité. **Complémentaire à toute solution technique** — sans cette famille, aucune solution tech ne survit (P1.4 : "le KM meurt par le coût de contribution").
    
- 🔵 **Famille B — "Outils du marché (Buy)"** : Options #1, #6, #7, #9 → _Vitesse d'implémentation maximale._ On trade l'ambition long-terme contre la rapidité et la réduction du risque. **BookStack (#1) est le chemin de moindre résistance absolu.**
    
- 🟣 **Famille C — "Sur-mesure (Build)"** : Options #10, #12, #14, #23 → _Fit parfait mais coûteux/risqué._ Spectre complet : du #10 (scope castré, risque moindre, différenciation faible) au #14 (scope complet, modèle irréaliste). **Le #23 est le point d'équilibre validé par le sponsor : scope complet du #14, modèle de réalisation réel (Alexandre + Claude Max), discipline du séquençage en 3 jalons.** C'est l'option cible du projet.
    
- 🟠 **Famille D — "Fork & Extend"** : Options #8, #11 → _Le meilleur des deux mondes._ Effort divisé par 5-10x vs from scratch. **C'est le "plan B assumé" de P1.4 — le filet de sécurité si le #23 échoue au spike.**
    
- 🔴 **Famille E — "IA & Automatisation"** : Options #13, #15, #16, #22 → _Changement de paradigme._ **Combinable avec n'importe quelle famille ci-dessus comme couche d'augmentation.**
    
- 🟤 **Famille F — "Fédération sans migration"** : Options #2, #5, #12 → _Ne rien migrer, tout fédérer._ **Option de transition idéale pendant le développement d'une solution cible.**
    
- ⚪ **Famille G — "Culture & Communauté"** : Options #17, #18, #20, #21 → **Plus pertinent comme philosophie de design que comme solution autonome.**

## 8. Combinaisons Recommandées (Pour inspiration P3.2)

Les options ne sont pas mutuellement exclusives. Les combinaisons les plus intéressantes :

**Combo "Quick Win + Vision" (RECOMMANDÉ)** : #1 (BookStack Full Send) + #4 (Concierge Humain) + #19 (Tinder Fraîcheur) **maintenant** → **#23 (PlumeNote Vision Séquencée) en parallèle** dès S+1, avec gate review à S+5. BookStack délivre la valeur immédiate et constitue la base de contenu. PlumeNote se construit en arrière-plan. Bascule à M+3 quand le jalon 1 est production-ready. Si le spike échoue → #8 (Fork Augmenté Docmost) comme plan B intermédiaire.

**Combo "Fédérer puis Migrer"** : #2 (Loupe Magique) + #5 (Wrapper Chrome) **immédiatement** → #23 (PlumeNote Vision Séquencée) ou #7 (Docmost) **en parallèle**. La recherche fédérée couvre le besoin #1 pendant que la solution cible mûrit.

**Combo "IA-First"** : #13 (Cerveau DSI / RAG) + #15 (Auto-Doc depuis GLPI) + #22 (Radio DSI). On ne construit pas un wiki — on construit un assistant qui exploite la documentation existante et en génère de nouvelle automatiquement. Paradigme radicalement différent.
